## Overview
This project contains the **receiver / gateway firmware** for a post-quantum secure communication demo on the **Bouffalo Lab BL602 / PineCone** platform.

The receiver:
- connects to Wi-Fi
- loads an existing **ML-KEM-512** key pair from internal flash, or generates and stores a new one
- listens for **CoAP over UDP** on port **5683**
- returns its public key when `/pqkem-pk` is requested
- receives one protected `/pqkem-data` message from the sender
- performs **ML-KEM decapsulation**
- derives an **AES-128** session key using **HKDF-SHA-256**
- decrypts and authenticates the payload using **AES-CCM**
- displays the plaintext on an **SSD1306 OLED**
- indicates success or failure using **GPIO5 LED**, OLED text, and serial logs
- detects **replay** and **authentication failure** events



## Features
- BL602 Wi-Fi station mode
- CoAP over UDP on port **5683**
- **ML-KEM-512** long-term key pair stored in internal flash
- **ML-KEM decapsulation** for shared-secret recovery

- Replay detection using a nonce cache
- Minimal CoAP implementation for embedded use
- SSD1306 OLED message display over I2C
- GPIO5 LED indication for success and failure

The project includes a helper script named `genromap`. Build from the project root:

```bash
cd Receiver
./genromap
```

This starts the compilation process for the BL602 target.

## Flashing the Board
The flashing steps below follow the same BL602 compile-and-flash flow used for the sender project.

### 1. Put the PineCone into flashing mode
- Bridge **IO8** with **H**
- Press the **reset** button

### 2. Flash the compiled firmware
After a successful build, flash the generated binary from `build_out/`.

#### GNU/Linux
```bash
blflash flash build_out/Receiver.bin --port /dev/ttyUSB0
```

If your BL602 appears on another serial device, replace `/dev/ttyUSB0` with the correct port.

#### Windows
```bash
blflash flash build_out\Receiver.bin --port COM5
```

Replace `COM5` with the actual port shown in Device Manager.

> Depending on SDK naming, the output file may appear with a slightly different case or project-name variant inside `build_out/`. Use the receiver firmware binary generated by your build.

### 3. Return the PineCone to normal operating mode
- Bridge **IO8** with **L**

### 4. Open the serial console (optional but recommended)
#### GNU/Linux
```bash
screen /dev/ttyUSB0 2000000
```

#### Windows
Use a serial terminal and set the baud rate to:

```text
2000000
```

### 5. Reset the board
Press the **reset** button again to start the receiver firmware.

## Running the System

### Terminal 1 – Receiver
Flash and start this receiver firmware on one BL602 board.

### Terminal 2 – Sender
Flash and start the matching sender firmware on a second BL602 board.

### Expected run sequence
1. Receiver connects to Wi-Fi
2. Receiver loads or generates its ML-KEM key pair
3. Receiver starts listening on UDP port **5683**
4. Sender requests `/pqkem-pk`
5. Receiver returns the public key
6. Sender sends one protected `/pqkem-data` message
7. Receiver decapsulates, derives the AES key, authenticates, decrypts, and displays the plaintext

## Runtime Flow
The receiver implementation in `Receiver/Receiver/main.cpp` performs this sequence:

1. Initialize platform services, Wi-Fi, TCP/IP stack, and LED GPIO
2. Wait until Wi-Fi and DHCP are ready
3. Start loop runtime support required for the OLED driver
4. Initialize the SSD1306 OLED and show **"Waiting..."**
5. Load the stored ML-KEM key pair from flash, or generate and save a new one
6. Open a UDP socket and bind to port **5683**
7. Parse incoming CoAP messages
8. On `POST /pqkem-pk`, return the receiver public key
9. On `POST /pqkem-data`, validate lengths and message type
10. Check the nonce cache to reject replayed packets
11. Perform **ML-KEM decapsulation** to recover the shared secret
12. Derive the AES key using **HKDF-SHA-256** with `"ML-KEM-AEAD"`
13. Decrypt and authenticate the message with **AES-CCM**
14. On success, display the plaintext on the OLED and blink the LED once
15. On replay or auth failure, show the error on the OLED and blink the LED with an error pattern

## CoAP Endpoints

### Public key service
- **Method:** `POST`
- **Path:** `/pqkem-pk`
- **Port:** `5683`
- **Purpose:** return the receiver ML-KEM public key to the sender

### Protected data service
- **Method:** `POST`
- **Path:** `/pqkem-data`
- **Port:** `5683`
- **Purpose:** receive KEM ciphertext + AEAD-protected payload from the sender

## Message Format
Inside the `/pqkem-data` CoAP payload, the receiver expects a packed structure with:

- `type` – message type
- `reserved` – reserved byte
- `kem_ct_len` – ML-KEM ciphertext length
- `text_len` – protected plaintext length
- `nonce[12]` – AES-CCM nonce
- `buf[...]` – concatenation of:
  - KEM ciphertext
  - AES-CCM ciphertext
  - AES-CCM authentication tag

Current receiver constants:
- AES key length: **16 bytes**
- Nonce length: **12 bytes**
- Tag length: **16 bytes**
- Maximum plaintext length: **64 bytes**
- UDP buffer: **1024 bytes**
- Replay cache size: **32 nonces**

## Hardware Notes

### SSD1306 OLED
The receiver initializes the OLED through I2C.
The included I2C driver configures:
- **SDA = pin 3**
- **SCL = pin 4**

### Status LED
- **GPIO5** is used as the receiver status LED
- One long blink indicates a successful decrypt
- Short repeated blinks indicate replay or authentication failure

## Troubleshooting

### Build fails
- Check that `BL60X_SDK_PATH` is set
- Activate the correct Python virtual environment
- Verify the BL602 toolchain installation

### Flash fails
- Confirm the board is in **flashing mode** using **IO8 ↔ H**
- Check the correct serial port
- Make sure no other terminal is already using the port

### No Wi-Fi connection
- Recheck `wifi.h`
- Ensure the access point is reachable
- Confirm SSID and password are correct

### Sender does not get a response
- Make sure the receiver firmware is running
- Ensure both boards are on the same network
- Confirm UDP port **5683** is not blocked
- Check that the sender is requesting `/pqkem-pk`

### OLED does not display text
- Verify SDA/SCL wiring
- Check power and ground connections
- Confirm the OLED is SSD1306-compatible

### Receiver shows AUTH FAIL
- The message was modified, the session key does not match, or the sender/receiver builds are not compatible
- Make sure both sides use the same ML-KEM variant and message format

### Receiver shows REPLAY DROP
- The same nonce was sent again
- Restart the sender with a fresh encrypted packet

## Security Notes
- The receiver stores its long-term ML-KEM key pair in internal flash memory
- The shared secret is recovered through **ML-KEM decapsulation**
- The AES key is derived through **HKDF-SHA-256** rather than using the raw shared secret directly
- **AES-CCM** provides confidentiality and integrity for the protected payload
- Replay checking is implemented using a nonce cache
